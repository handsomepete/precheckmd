# HOME-447 â€” Wiring `cowork.main` to the new tools

This is a wiring spec, not a design doc â€” everything here is meant to be
followed directly on the Hetzner box in one sitting, with no need to read
`mcp/server.js` first. All example JSON below was generated by actually
running the relevant code (not hand-typed), so it's exact.

## Prerequisite (already done, this repo)

`mcp/server.js` (the `homeos-mcp` / "PersonalMCP2" server, `pm2 restart
homeos-mcp` to pick up a deploy of this branch) now exposes three new tools
and one extended tool over the same MCP endpoint cowork.main already talks
to: `POST {MCP_BASE_URL}/mcp`, JSON-RPC 2.0, method `tools/call`.

**`POST /mcp` now requires the same bearer auth as everything else** (see
SECURITY.md) â€” if cowork.main doesn't already send
`Authorization: Bearer <BEARER_TOKEN>` on its `/mcp` calls, add that header
first or nothing below will work. If cowork.main was relying on `/mcp`
being open, this is the one part of this wiring that isn't purely additive â€”
check that first.

## 1. `get_upcoming_bills` â€” replaces the stale-amount FINANCE bill lines

**Replaces:** whatever cowork.main currently does with
`get_scheduled_transactions` output to build the "due in N days" bill lines
in the FINANCE section. Stop reading `get_scheduled_transactions` for this
purpose; read `get_upcoming_bills` instead. (`get_scheduled_transactions`
itself is unchanged and still works for anything else that needs raw
scheduled-transaction data.)

**Request:**
```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": { "name": "get_upcoming_bills", "arguments": {} }
}
```
No arguments.

**Response** (`result.content[0].text`, JSON-parsed):
```json
{
  "bills": [
    {
      "id": "b3f1",
      "payee": "Hydro One",
      "status": "paid",
      "scheduled_amount": -426,
      "actual_amount": -454,
      "cleared_date": "2026-07-07",
      "matched_transaction_id": "t771",
      "due_date": "2026-07-10"
    },
    {
      "id": "c9a2",
      "payee": "Netflix",
      "status": "due",
      "amount": -20,
      "due_date": "2026-07-22",
      "days_until_due": 14
    }
  ]
}
```

**Rendering rule:** for each item in `bills`:
- `status: "paid"` â†’ render as paid, using `actual_amount` (not
  `scheduled_amount`) and `cleared_date`. This is the HOME-447 fix â€” do not
  print `scheduled_amount` as if it were owed. Example line: `Hydro One:
  paid $454.00 (cleared 2026-07-07)`.
- `status: "due"` â†’ render as due, using `amount` and `days_until_due`.
  Example line: `Netflix: $20.00 due in 14 day(s) (2026-07-22)`.

All amounts are dollars, negative = outflow (same sign convention as every
other YNAB tool in this server â€” no change there).

**On tool error** (`isError: true` in the MCP response, or the HTTP call
itself fails): fall back to the old behavior â€” render from
`get_scheduled_transactions` with a "due in N days" line and no
reconciliation, exactly as today. This is strictly worse (the original bug
can recur) but strictly better than the brief failing outright. Log the
error so it surfaces the next time someone looks, the same way HA/Linear
failures already do via `get_health`.

## 2. `get_property_status` â€” replaces the per-entity Weather/Garage calls

**Replaces:** separate `get_home_state` calls (or however cowork.main
currently gets weather + garage/cover data) for the PROPERTY section. Stop
making two+ separate HA calls for weather and garage; call this once
instead. (`get_home_state` is unchanged and still works for anything else,
e.g. controlling lights.)

**Request:**
```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "tools/call",
  "params": { "name": "get_property_status", "arguments": {} }
}
```
No arguments.

**Response, HA reachable:**
```json
{
  "ok": true,
  "weather": [
    {
      "entity_id": "weather.home",
      "state": "cloudy",
      "attributes": { "temperature": 21.5, "humidity": 60, "friendly_name": "Home" },
      "last_changed": "2026-07-08T11:00:00+00:00"
    }
  ],
  "covers": [
    {
      "entity_id": "cover.garage_door",
      "state": "closed",
      "attributes": { "friendly_name": "Garage Door" },
      "last_changed": "2026-07-08T07:15:00+00:00"
    }
  ]
}
```
`weather`/`covers` are raw Home Assistant `/api/states` entities (full
`attributes` object, not trimmed) â€” same shape you'd get from
`get_home_state`, just pre-filtered to the `weather.*`/`cover.*` domains from
one HA call.

**Response, HA unreachable (the actual HOME-447 bug case):**
```json
{
  "ok": false,
  "line": "Property status unavailable (HA unreachable)",
  "error": "Client network socket disconnected before secure TLS connection was established"
}
```

**Rendering rule:** check `ok` first.
- `ok: true` â†’ render weather + garage/cover lines from `weather`/`covers` as
  today.
- `ok: false` â†’ render **only** `line` as the entire PROPERTY section. Do
  **not** print `error` in the brief body (it's operational detail, not
  something you need every morning) â€” it's there for logs/debugging only.
  This is the fix: one line instead of a raw socket error per entity.

**On tool error** (the MCP call itself errors, distinct from `ok: false`
which is a normal response): treat it the same as `ok: false` â€” render the
same `"Property status unavailable (HA unreachable)"` line. There's no
useful distinction for the brief reader between "the tool told us HA is
down" and "we couldn't even call the tool."

## 3. `get_linear_focus` â€” replaces the raw "Entity not found" crash

**Replaces:** however cowork.main currently resolves "TODAY'S LINEAR FOCUS"
(reading some pinned/cached issue id and fetching it directly). Call this
instead, passing whatever id is currently pinned/cached.

**Request:**
```json
{
  "jsonrpc": "2.0",
  "id": 3,
  "method": "tools/call",
  "params": {
    "name": "get_linear_focus",
    "arguments": { "pinned_issue_id": "a1b2c3d4-e5f6-4789-a012-3456789abcde" }
  }
}
```
`pinned_issue_id` is optional â€” omit it entirely to skip straight to the
highest-priority-open-issue fallback (e.g. if there's no longer a concept of
a "pinned" issue on the cowork.main side).

**Response, pinned issue no longer exists (the actual HOME-447 bug case):**
```json
{
  "issue": {
    "id": "HOME-451",
    "title": "Renew HA long-lived token",
    "priority": 1,
    "due": "2026-07-08",
    "url": "https://linear.app/homestead/issue/HOME-451"
  },
  "source": "fallback",
  "dangling_issue_id": "a1b2c3d4-e5f6-4789-a012-3456789abcde"
}
```

**Response, pinned issue exists:** same `issue` shape, `"source": "pinned"`,
`"dangling_issue_id": null`.

**Response, nothing to show:** `"issue": null`, `"source": "none"`.

**Rendering rule:**
- `issue: null` â†’ render `"No open urgent issues."` or omit the section.
- otherwise â†’ render `issue.title` (already sanitized server-side â€” see
  SECURITY.md â€” safe to render as plain single-line text) with `issue.id`.
- **When `source: "fallback"`**, `dangling_issue_id` is non-null: this is
  exactly the id that no longer resolves in Linear. **Append a short note to
  the brief** (e.g. `(pinned issue ${dangling_issue_id} no longer exists â€”
  clean up this reference)`) or at minimum log it, so it actually gets
  cleaned up instead of silently falling back forever. The tool already logs
  a warning server-side (`[linear-focus] pinned issue "..." no longer
  exists...`), but that log lives on the MCP server, not wherever
  cowork.main's own logs go â€” don't rely on it being seen from the
  cowork.main side.

**On tool error:** render `"TODAY'S LINEAR FOCUS: unavailable"` â€” do not let
a raw error string reach the brief (this was the literal HOME-447 symptom:
`"Error: Entity not found: Issue"` printed directly).

## 4. `create_gmail_draft` â€” pass `brief_type` explicitly

**Change needed in cowork.main:** wherever it currently calls
`create_gmail_draft` to send the finished brief, add one field:

```json
{
  "jsonrpc": "2.0",
  "id": 4,
  "method": "tools/call",
  "params": {
    "name": "create_gmail_draft",
    "arguments": {
      "to": "you@example.com",
      "subject": "Morning Brief â€” 2026-07-08",
      "body_html": "<html>...</html>",
      "body_text": "...",
      "brief_type": "morning"
    }
  }
}
```

`brief_type` must be exactly `"morning"` or `"evening"` â€” matching whichever
`run_brief` flag triggered this run. That's the only change on the
cowork.main side for this part.

**Why:** today, `create_gmail_draft` guesses whether a draft is a brief (and
which type) by regexing the subject line (`/morning brief/i` etc. â€” see
`mcp/lib/briefStore.js`). That's a stopgap that works but is fragile (a
subject rewrite silently stops the file dual-write with no error). Passing
`brief_type` explicitly makes the dual-write to
`briefs/YYYY-MM-DD-{brief_type}.md` (servable via `GET /briefs/:filename`,
auth-gated â€” see SECURITY.md) unconditional and correct, regardless of
subject wording.

**Once cowork.main passes `brief_type` on every brief-creating call**, file
a follow-up to delete `detectBriefType()` and its subject-matching in
`mcp/lib/briefStore.js` (leave `maybePersistBriefFromDraft`'s
`brief_type`-first logic in place, just remove the now-dead fallback path
and its tests). Do not delete the heuristic before confirming cowork.main
is actually passing `brief_type` on every call â€” until then it's still the
only thing making the dual-write work.

**Behavior if you don't make this change:** nothing breaks. The subject-line
heuristic keeps working exactly as it does today. This section is purely
optional hardening, not a blocker for using items 1â€“3 above.

## Verifying the wiring worked

After redeploying cowork.main with the above:

```bash
# Confirm the file landed (replace host/token with the real ones):
curl -s -H "Authorization: Bearer $BEARER_TOKEN" https://<mcp-host>/briefs
curl -s -H "Authorization: Bearer $BEARER_TOKEN" https://<mcp-host>/briefs/2026-07-08-morning.md
```

If `GET /briefs` comes back empty after a real brief run, `brief_type` isn't
reaching `create_gmail_draft` (or the subject-line heuristic didn't match,
if you haven't made change #4 yet) â€” check the MCP server logs for
`[briefStore] failed to persist brief file: ...`, which would indicate a
different problem (e.g. `BRIEFS_DIR` not writable).
